# Интеграция с 1С через MCP Toolkit

Шлюз получает остатки из **1С:ERP** через сторонний MCP-сервер
[**1C MCP Toolkit**](https://github.com/ROCTUP/1c-mcp-toolkit) (автор ROCTUP, лицензия GPL-3.0)
посредством его REST `/api/execute_query`. Это не потребовало изменения конфигурации 1С
и публикации 1С на веб-сервере. Наш код только обращается к REST API тулкита по сети —
отдельной GPL-зависимости в репозитории нет.

## Установка обработки MCP_Toolkit.epf в 1С

МCP-сервер живёт **внутри сеанса внешней обработки** 1С. Порядок запуска:

1. Скачай готовую обработку из репозитория тулкита:
   [build/MCP_Toolkit.epf](https://github.com/ROCTUP/1c-mcp-toolkit/blob/main/build/MCP_Toolkit.epf)
   (актуальную версию можно взять на странице [Releases](https://github.com/ROCTUP/1c-mcp-toolkit/releases)).
2. Запусти нужную базу 1С (тип конфигурации: 1С:ERP / Комплексная автоматизация / Управление торговлей —
   где есть регистр `ТоварыНаСкладах`). Это **обычный** сеанс 1С (толстый/тонкий клиент) под пользователем
   с правом на чтение регистра и запуск внешних обработок.
3. **Файл → Открыть…** → укажи скачанный `MCP_Toolkit.epf`.
4. В форме обработки выбери режим **«Встроенный сервер»** (вариант без Python — HTTP-сервер поднимается прямо в 1С).
5. При необходимости задай порт (по умолчанию `6003`) и ID канала изоляции (опц., см. `ONEC_CHANNEL`).
6. Нажми **«Запустить сервер»**. В окне сообщений должна появиться строка:
   `Встроенный HTTP-сервер запущен на порту 6003`.
7. Не закрывай обработку, пока нужен голосовой цикл: остановка сеанса/закрытие обработки гасит сервер.

Проверка запуска (из PowerShell на хосте, где крутится 1С):
```powershell
netstat -ano | Select-String ':6003'    # -> TCP 0.0.0.0:6003 LISTENING <pid>
./q1c.ps1 -Query 'ВЫБРАТЬ 1'            # -> success=True
```

> **Файервол.** Подключения с `localhost` и из Docker (`host.docker.internal`) работают без правил.
> Если 1С на другой машине — добавь входящее правило Windows Firewall для TCP `6003` и используй
> её IP в `ONEC_BASE_URL` (например `http://192.168.0.50:6003/api`).
>
> **Каналы.** Несколько баз/сеансов 1С можно изолировать параметром `?channel=<id>` — задай тот же ID
> в обработке и в `ONEC_CHANNEL`.

## Транспорт

- 1C MCP Toolkit запускает встроенный HTTP-сервер **внутри обработки** 1С на порту `6003`
  (слушает `0.0.0.0:6003`).
- REST-база: `http://host.docker.internal:6003/api` (из контейнеров), на хосте — `http://127.0.0.1:6003/api`.
- Эндпоинт: `POST /api/execute_query`, body `{"query": "<запрос 1С>", "limit": 50}`.
- Ответ: `{"success": true, "data": "<значение-таблица текстом>"}`.
- Изоляция каналов: `?channel=<id>` (опционально, `ONEC_CHANNEL`).

## Запрос остатков

Регистр `РегистрНакопления.ТоварыНаСкладах.Остатки` (виртуальная таблица остатков):
- измерения: `Номенклатура`, `Склад`, `Помещение`, `Характеристика`, `Серия`, …
- ресурс остатка: **`ВНаличииОстаток`** (+ `КОтгрузкеОстаток`).

Шлюз группирует сумму `ВНаличииОстаток` по складам и фильтрует номенклатуру по
названию (`ПОДОБНО` без учёта регистра через `ВРЕГ`):

```1c
ВЫБРАТЬ
  ТоварыНаСкладахОстатки.Склад.Наименование КАК Склад,
  СУММА(ТоварыНаСкладахОстатки.ВНаличииОстаток) КАК Остаток
ИЗ РегистрНакопления.ТоварыНаСкладах.Остатки КАК ТоварыНаСкладахОстатки
ГДЕ ТоварыНаСкладахОстатки.ВНаличииОстаток <> 0
  И ВРЕГ(ТоварыНаСкладахОстатки.Номенклатура.Наименование) ПОДОБНО ВРЕГ("%<товар>%")
СГРУППИРОВАТЬ ПО ТоварыНаСкладахОстатки.Склад.Наименование
УПОРЯДОЧИТЬ ПО Остаток УБЫВ
```

Ввод товара sanitizится (только буквы/цифры/пробелы/дефис) — безопасно для `ПОДОБНО`.

## Формат ответа и парсер

Тулкит возвращает значение-таблицу своим текстовым форматом:
```
[5]{"Склад","Остаток"}:
  Торговый зал,150
  Центральный склад,100
  "Магазин \"Продукты\"",100
  ...
```
- `[N]` — число строк; `{...}` — имена колонок; `:` завершает заголовок.
- Пустой результат: `[0]:`.
- Строки со спецсимволами — JSON-заключены (`\"`), числа — как есть.

Парсер `voice-gateway/onec.py::_parse_table` разбирает заголовок и строки,
переводя строковые поля через `json.loads`, числа — в `int/float`.

> Ограничение: парсер рассчитан на колонки типа «строка/число». Не возвращайте
> `objectRef` напрямую — соберите нужные поля через `.Наименование`.

## Контракт остатка (внутренний)

`call_stock_api(item)` → `dict`:
```json
{
  "item": "молоко",
  "found": true,
  "quantity": 480,
  "warehouses": [{"name": "Торговый зал", "quantity": 150}, …],
  "message": "Остаток 'молоко': всего 480 единиц. По складам: …",
  "source": "1c"
}
```
`source` бывает `1c`, `mock`, `mock(fallback: <Error>)`.

## Ручные проверки

```powershell
# из хоста (UTF-8 обязательно для кириллицы)
./q1c.ps1 -Query 'ВЫБРАТЬ ПЕРВЫЕ 5 Наименование ИЗ Справочник.Склады'

# из контейнера (как видит шлюз)
docker run --rm --add-host=host.docker.internal:host-gateway curlimages/curl:latest `
  -s -X POST "http://host.docker.internal:6003/api/execute_query" `
  -H "Content-Type: application/json" --data-binary '@q.json'
```

## Известные нюансы

- **Единицы измерения**: `ВНаличииОстаток` считается в единице хранения каждой
  номенклатуры. Сумма по разным позициям (напр. «сахар» весовой в кг и в пачках)
  смешивает единицы — для голосового MVP допустимо, при необходимости группируйте
  по `ЕдиницаИзмерения` / Номенклатуре.
- **Доступность**: при `STOCK_FALLBACK_TO_MOCK=true` и ошибке 1С ответ приходит из
  mock-api, в `source` будет `mock(fallback: …)` — видно в `/health` и логах шлюза.
